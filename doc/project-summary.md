# Agent 工具调用授权：基于可信意图上界与运行时一致性验证的预执行防护

## 一、课题描述

随着大语言模型（LLM）智能体逐步从“对话生成”走向“调用工具并操作外部环境”，其能力边界正在快速扩展到邮件发送、数据查询、文件修改、系统控制等真实业务流程之中。在这一背景下，工具调用已不再只是模型输出的一段结构化文本，而是可能直接作用于真实资源、影响真实用户、触发真实后果的执行行为。因此，工具调用安全的关键不在于函数参数是否格式正确，而在于系统能否在执行前，对该调用是否符合用户可信任务目的作出可解释、可约束、可审计的判断。

与传统内容安全问题相比，智能体工具调用的风险更具执行性和现实危害性。模型可能在恶意请求、提示注入或复杂上下文诱导下发起越权访问、敏感信息外发、错误对象操作或其他不可逆行为；而这类调用在参数表面上往往仍然“格式正确”，仅凭参数结构本身并不足以判断其真实意图、任务必要性及潜在受影响方。与此同时，现有 Function Calling 机制主要关注参数格式与数据结构约束，现有防护方法则多依赖事前提示词约束、静态规则过滤或事后异常检测，缺乏在工具真正执行前进行步骤级、主动式和细粒度干预的能力。

基于此，本课题聚焦于“面向真实任务目的的基于意图的访问控制（Intent-Based Access Control）与安全授权”。核心思路是在 function call 机制中引入结构化意图参数，使模型同步生成调用意图。更关键的是，针对模型在受攻击时可能生成“虚假意图”以掩盖恶意行为的隐蔽风险，本课题引入了多维交叉验证机制：在工具真正执行前，外挂式护栏中间件中的护卫模型不仅依赖本次声明的结构化意图，还会强制联合原始用户指令、历史 function call 轨迹以及当前函数参数进行一致性推理。通过识别“伪造意图”与“上下文轨迹/实际参数”之间的逻辑矛盾，护卫模型能够精准识破意图欺骗，并对调用作出放行、拒绝、重写、缩权、确认或替换等细粒度决策，同时保留审计记录。

## 二、问题脉络、已有方法不足与本课题切入点

### 1. 问题脉络

当前智能体工具调用安全的核心矛盾在于：模型已经具备调用真实工具并作用于外部环境的能力，但系统对一次调用“为什么发生、是否必要、会影响谁、风险是否可接受”的分析理解能力仍然不足。也就是说，风险判断不能只停留在参数合法性层面，必须向基于意图的访问控制（Intent-Based Access Control）演进，进一步识别其任务目的及上下文必要性。同时，必须正视意图本身的可靠性问题——在意图可能被伪装或篡改的复杂攻击场景下，系统仍需具备穿透虚假声明、还原真实执行逻辑的兜底判别能力。

### 2. 已有方法不足

尽管近期部分前沿工作在智能体安全防护方面做出了重要探索，但面对真实复杂的工具调用场景，仍存在以下显著局限：

缺乏意图深度的步骤级护栏：例如 ToolSafe (2026) 等研究做出了重要贡献，开始引入主动的步骤级护栏与反馈机制（Step-level Guardrail and Feedback），尝试在工具执行前进行拦截。然而，这类方法大多仍依赖对“表面参数”的异常检测，缺乏对调用背后真实目的和受影响方的结构化理解。此外，其依赖的反复反馈重试机制（Feedback loop）容易显著增加系统延迟，难以满足真实业务链条对低延迟的工程要求。

过于静态与二元的意图访问控制：业界也开始探索从传统的“基于动作的权限”向“基于意图的访问控制（IBAC）”转型，例如 Potti (2026) 提出的细粒度授权方案。但在实际应用中，现有 IBAC 方法多依赖于将用户初始指令外部解析为静态规则（如 FGA 元组），并进行确定性拦截。这种静态解析方式无法适应智能体在复杂环境中多步推理时动态变化的运行时意图（Runtime Intent）；同时，简单的“放行/拒绝（Allow/Deny）”二元逻辑极大地损害了 Agent 的自主任务完成率。

“意图欺骗”的脆弱性与交叉验证缺失：现有试图提取意图的方案（包括上述前沿工作）往往预设“模型生成的意图或解析出的指令即为真实意图”，忽略了攻击者可以通过复杂上下文诱导智能体生成“表面合规的虚假意图”来合理化恶意动作。由于缺乏将当前声明意图与原始指令、历史轨迹、真实参数进行多维交叉验证的机制，导致现有防护极易被高级隐蔽攻击绕过。

| 代表工作 | 主要思路 | 启发 | 仍然存在的缺口 |
|---|---|---|---|
| CaMeL | 从可信 query 中提取控制流/数据流，将不可信数据与高权限执行隔离。 | 说明“可信输入边界”必须先于工具调用安全判断被定义。 | 更强调结构隔离，对“声明意图与实际调用不一致”这一运行时欺骗问题刻画不足。 |
| IPIGuard | 通过工具依赖图将动作规划与外部数据交互解耦。 | 说明“先规划、后执行”的结构化设计能降低注入影响。 | 更偏执行范式改造，尚未把“意图声明失真”上升为授权对象。 |
| ToolSafe | 在执行前做步骤级 guardrail，对高风险工具调用进行检测与反馈。 | 说明执行前干预、步骤级审查是正确方向。 | 重点仍在动作风险与反馈纠偏，对“表面合规但实质越权”的意图欺骗识别不够专门。 |
| OASIS | 把意图隐藏与任务复杂度引入 Agent 安全评测。 | 说明安全性会因“恶意意图被稀释、隐藏”而显著下降。 | 更偏评测与现象揭示，没有直接给出运行时授权机制。 |
| IBAC | 从用户显式意图导出细粒度权限，并在工具调用前做确定性检查。 | 说明“从意图导出权限边界”非常有价值。 | 偏静态授权映射，难以覆盖多步任务中运行时语义漂移与声明—执行不一致。 |

### 3. 本课题切入点

针对上述前沿方法的不足，本课题选择从“调用协议增强 + 多维交叉比对 + 细粒度干预 + 动态攻防训练”四个层面切入：

调用协议增强（突破参数异常检测的局限）：在 function call 参数协议中直接嵌入结构化意图，将 action type、target object、goal、necessity 等安全语义显式化，赋予护栏深度的意图理解能力。

多维交叉护栏防欺骗（突破意图欺骗的脆弱性）：部署外挂式执行前护栏中间件。护卫模型不再单一轻信生成的意图，而是联合用户原始指令、历史调用轨迹、当前工具参数进行深度逻辑比对。一旦发现生成的意图与上下文历史或工具行为存在矛盾，护卫模型将精准识破意图欺骗。

细粒度动态干预机制（突破静态与二元控制的局限）：护卫模型在执行前完成步骤级授权时，突破传统的 Allow/Deny 二元逻辑，输出重写、缩权、确认或替换等更贴合运行时环境的动态细粒度控制动作，在保障安全的同时最大化维持智能体的任务完成率。

动态攻防驱动的模型训练（突破静态规则与训练数据瓶颈）：护卫模型的训练数据并非人工静态编写，而是基于 AgentDojo、InjecAgent、LLMail-Inject 等已有攻击数据构造：将攻击数据作为输入喂给合适的智能体，在多个不同 LLM 底座下生成调用过程，再抽取并标注“意图—参数—风险—决策”样本，用于训练护卫模型基座。这样，系统的判断依据将从单纯的参数表面，提升为面向真实任务目的、潜在影响对象与风险后果的综合推理。

## 三、本课题的创新点

1. 提出防意图欺骗的多维交叉验证机制（Cross-Validation against Intent Deception）：针对智能体在受攻击或复杂上下文中可能生成“表面合规的虚假意图”这一隐蔽风险，突破了传统意图识别“轻信模型输出”的盲区。护卫模型不单一依赖本次声明的意图，而是强制联合原始用户指令、历史 function call 轨迹以及当前函数参数进行一致性推理。即使意图被刻意伪装，系统也能通过捕捉“伪造意图”与“上下文轨迹/实际执行动作”之间的逻辑矛盾，精准识破意图欺骗并实施拦截。

2. 构建基于意图的访问控制（Intent-Based Access Control）与协议层增强：在 function call 协议层引入结构化意图参数，将原本隐含在模型推理过程中的调用目的、目标对象、必要性、影响对象与风险提示显式化。这使安全判断从传统的“只看参数格式”推进到“同时理解调用语义与任务目的”，为实现细粒度授权（Fine-Grained Authorization）提供了底层的协议级支持。

3. 引入主动式步骤级护栏（Proactive Step-level Guardrail）与细粒度干预机制：提出外挂式执行前护栏机制，并以护卫模型作为核心判定单元。在工具真正执行前完成步骤级授权与细粒度干预，突破传统 allow/deny 二元控制，支持重写、缩权、替换与确认等更贴合智能体场景的干预动作。

4. 打造动态攻防驱动的护卫模型训练流水线：将护卫模型视为可训练的安全模型而非静态规则器。基于 AgentDojo、InjecAgent、LLMail-Inject 等攻击数据构造训练流水线，并专门针对“意图欺骗”场景构造包含上下文矛盾的复杂对抗样本。通过不同智能体与不同 LLM 底座生成多样化调用过程，形成监督样本，弥补执行前护栏训练数据不足的问题。

5. 建立面向真实任务目的的可靠授权与可追溯审计闭环：将结构化意图、调用参数、历史轨迹、策略约束和审计记录纳入统一框架。结合多维交叉比对的结果，形成一套面向真实任务目的的可靠授权与可追溯审计机制，兼顾安全性、可解释性与工程可部署性。

## 四、研究思路、数据集与参考文献

### 1. 研究思路

- 第一步，设计结构化意图 schema，并将其嵌入 function call 参数协议，形成统一的调用表示。
- 第二步，构建执行前护栏中间件：输入包括结构化意图、工具参数、历史轨迹和策略规则；输出包括放行、拒绝、重写、缩权、确认、替换等动作。
- 第三步，搭建护卫模型数据构造流水线：以 AgentDojo、InjecAgent、LLMail-Inject 等数据集中的攻击样本和任务上下文作为输入，选择合适的智能体并结合不同 LLM 底座生成工具调用过程，进一步抽取和标注“意图—参数—风险—决策”样本数据。
- 第四步，基于上述样本训练护卫模型基座，并开展离线/在线评测，重点考察攻击成功率、越权率、误报率、额外延迟、任务完成率、泛化能力和审计可用性。
- 第五步，结合策略引擎与审计日志，验证该方案在真实 agent 工具链中的兼容性、可解释性和可部署性。

### 2. 计划使用的数据集/基准

- AgentDojo：用于构造动态工具调用环境下的攻击与防护样本。
- InjecAgent：用于覆盖工具集成场景中的间接提示注入风险。
- LLMail-Inject：用于邮件智能体等真实业务场景中的自适应提示注入样本。
- ToolSafe/TS-Bench、Tensor Trust 等：用于补充评测场景与对照实验。
